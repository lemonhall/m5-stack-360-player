# PRD-0001：M5StickC Plus2 头部姿态传感器链路

## 愿景

把固定在眼镜上的 M5StickC Plus2 作为头部姿态传感器。第一阶段只完成传感器数据从 M5 到 PC 的稳定回传、实时显示、日志记录和 PC 端零点校准，为后续控制 360 VR 视角提供可验证的数据链路。

成功状态：

- M5StickC Plus2 屏幕上的官方 IMU/cube 可视化仍能正常显示。
- PC 能通过蓝牙接收 M5 发出的姿态和原始 IMU 数据。
- 用户按下 M5 主按钮后，PC 端把当前姿态记录为中心姿态，并显示相对姿态。
- 本阶段不控制 PotPlayer，不模拟鼠标拖拽。

## 背景

用户设备为橙黄色小棒形的 M5StickC Plus2。用户希望把它固定在眼镜上，把长方体一端视为观看者视线方向。由于设备随头部转动，后续可把相对姿态变化映射为 360 VR 播放器视角控制。

v1 的目标不是播放器控制，而是先把 M5StickC Plus2 当成透明传感器：保留官方 IMU/cube 示例，在其基础上新增蓝牙数据发送；PC 端接收、显示和记录数据。

## 术语

- **M5**：M5StickC Plus2。
- **PC**：Windows 11 主机，优先使用 PowerShell 和 Python 工具链。
- **姿态**：官方示例已有的 yaw/pitch/roll 或四元数；如果官方示例没有可直接导出的姿态，再补 Mahony/Madgwick 融合。
- **原始 IMU**：加速度计和陀螺仪读数。
- **中心姿态**：用户按下 M5 主按钮时，PC 端记录的当前姿态。
- **相对姿态**：当前姿态相对于中心姿态的差值或等价旋转。
- **遥测包**：M5 通过蓝牙发送给 PC 的一条 JSON 文本数据。

## 需求

### REQ-0001-001：固件基线必须来自官方 IMU/cube 示例

- **动机**：官方示例已经验证 IMU 初始化、姿态变化和屏幕可视化，直接复用可降低硬件调试风险。
- **范围**：先找到 M5StickC Plus2 对应官方 IMU/cube 示例源码，再决定原样导入或迁移到 PlatformIO/Arduino 工程。
- **非目标**：v1 不重写屏幕可视化，不主动替换官方 IMU 初始化流程。
- **验收口径**：工程中有明确来源记录；固件运行时屏幕仍显示随姿态变化的 cube 或等价官方可视化。

### REQ-0001-002：M5 必须发送姿态和原始 IMU 数据

- **动机**：姿态用于后续控制，原始 IMU 用于漂移、延迟和异常排查。
- **范围**：每个遥测包至少包含序号、M5 端毫秒时间戳、姿态、加速度、陀螺仪和按钮状态。
- **非目标**：v1 不追求二进制最小包，不做复杂压缩。
- **验收口径**：PC 日志中连续数据包包含 `seq`、`ms`、`acc`、`gyro`、`btn`，并包含 `ypr` 或 `quat` 中至少一种姿态字段。

### REQ-0001-003：蓝牙链路优先使用 BLE GATT 通知

- **动机**：BLE GATT 通知结构清晰，适合后续扩展字段和状态服务。
- **范围**：M5 作为 BLE 外设发布遥测通知；PC 优先用 Python `bleak` 订阅通知。
- **非目标**：v1 不把蓝牙串口 SPP 作为第一实现目标。
- **验收口径**：PC 端能发现并连接 M5 BLE 服务，收到遥测通知；若 BLE 受阻，计划中保留传输适配层以降级到串口或 USB。

### REQ-0001-004：遥测协议使用 JSON 文本

- **动机**：JSON 文本可直接打印、记录和人工检查，符合第一阶段透明调试目标。
- **范围**：M5 发送单条 JSON 文本；PC 端解析后写 JSONL 日志。
- **非目标**：v1 不使用自定义二进制协议。
- **验收口径**：PC 端保存的日志为 JSONL，每行可被 `json.loads` 独立解析。

### REQ-0001-005：采样/发送频率必须可配置

- **动机**：v1 默认低压力调试，后续可比较 20/30/60 Hz 的控制手感。
- **范围**：默认约 30 Hz，配置项允许调整到更低或更高频率。
- **非目标**：v1 不承诺 90 Hz 或硬实时延迟。
- **验收口径**：配置文件或固件常量中有明确发送频率；PC 日志能统计实际包间隔。

### REQ-0001-006：PC 端必须支持主按钮零点校准

- **动机**：设备固定在眼镜后的初始角度不可预设，需要用户把当前视线校准为屏幕正前方。
- **范围**：M5 只上传按钮事件；PC 端收到主按钮按下事件时记录中心姿态，并显示相对姿态。
- **非目标**：v1 不在 M5 固件中修改姿态数据为相对值。
- **验收口径**：按下 M5 主按钮后，PC 终端显示中心姿态已更新；随后相对 yaw/pitch/roll 或等价相对旋转从接近零开始变化。

### REQ-0001-007：PC 端必须同时实时显示和写 JSONL 日志

- **动机**：实时显示用于调试连接和佩戴姿态，JSONL 用于后续分析漂移、延迟和丢包。
- **范围**：终端显示连接状态、绝对姿态、相对姿态、原始 IMU、按钮状态、日志路径。
- **非目标**：v1 不做 GUI，不画 3D 预览，不控制 PotPlayer。
- **验收口径**：运行 PC 接收程序后，终端持续刷新数据；退出后日志文件存在且包含有效 JSONL 数据。

### REQ-0001-008：v1 明确排除 PotPlayer 控制

- **动机**：先验证传感器链路，避免同时调蓝牙、姿态和播放器控制造成边界不清。
- **范围**：只保留后续映射方向说明，不实现鼠标拖拽或窗口控制。
- **非目标**：v1 不调用 Windows 鼠标 API，不识别 PotPlayer 窗口，不改变播放器视角。
- **验收口径**：v1 代码和测试中不存在 PotPlayer 控制逻辑；文档只把它列为 v2 候选。

## 待决事项

- 固件工程最终采用 Arduino IDE 原工程、PlatformIO，还是从官方示例迁移后决定。
- 姿态字段优先发送 `ypr`、`quat` 或两者都发，需要在拿到官方示例后确认包长和数据来源。
